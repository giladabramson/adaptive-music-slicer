Pipeline

How it works

0

Generate

Your prompt hits a text-to-music backend — Google Lyria by default for best quality. MusicGen, Stable Audio, AudioLDM and Beatoven are drop-in alternatives.

--gen-backend lyria | musicgen | stable-audio | audioldm | beatoven
1

Separate

BS-Roformer-Viperx-1296 extracts the vocals stem (SDR ≈ 12.1, transformer on mel-spectrograms). Demucs htdemucs_ft — a 4-model fine-tuned ensemble — splits the residual instrumental into drums / bass / other.

BS-Roformer → htdemucs_ft → drums, bass, other, vocals
2

Lock to a loop

Librosa detects BPM and the first beat. The engine computes loop length from tempo (bars × 4 × 60 / BPM), never from two noisy beat detections — so loops are exact whole bars and don't drift.

BPM × bars → integer-ms loop window
3

Slice all four stems identically

All four stems get cut with the same loop_start_ms/loop_end_ms, quantised to integer milliseconds exactly once. Phase-locked. Sub-millisecond seam error.

4

Play adaptively

The runtime player drives one mix callback with a shared playhead. Whatever you wire to it — game state, a DJ knob, a UI toggle — fades stems on linear ramps. No crossfade gap, no audible re-trigger.

any input → per-stem gain → realtime mix

Questions

What you'll probably ask

Can I use this commercially?

Yes for the engine itself — it's MIT-licensed. Each generation backend has its own terms, though, and that's the part you need to read:

  • Lyria — Google's preview model. Read Google's preview terms before shipping commercial output; preview-tier outputs are often non-commercial.
  • MusicGen — Meta's research license. Check Meta's terms.
  • Stable Audio / AudioLDM / Beatoven — each has its own commercial tier. Pick the backend whose license fits your use case.

The separation, slicing, and runtime player are fully MIT — no strings on those.

Do I need a GPU?

No, but a GPU makes iteration usable. On CPU the hybrid takes ~200 s/song. On a laptop GPU (RTX 3050) ~30 s. On a T4 cloud GPU ~16 s. For final builds, CPU is fine. For development with re-rolls, you'll want the GPU.

Does this replace my DAW, Wwise/FMOD, or DJ software?

Different jobs, complementary tools.

  • DAWs (Ableton, Logic, Pro Tools) — StemForge can feed clean stems into them; it doesn't replace the arrangement/effects/mastering work a DAW is built for.
  • Wwise / FMOD — game-audio middleware for triggering and mixing audio you already have. StemForge generates the audio. Feed the stems into Wwise, or use the included 250-line runtime player directly.
  • DJ software (Serato, rekordbox) — StemForge can be the generator stage; the stems load directly into stem-aware DJ software, or play through the runtime player for live remix.
Do my players need to install Python?

No. Python is only for the build-time generation + slicing step. The runtime is just four WAV files and a config.json. The reference player is a ~250-line Web Audio module for HTML5 games, and the same logic ports cleanly to any engine's audio API (Unity, Unreal, Godot, Bevy — they all expose per-source gain control).

Is the audio quality good enough to ship?

Honest answer: depends on the game.

  • Indie, web, mobile, prototypes — often surprisingly good, especially Lyria-generated.
  • Atmospheric / background beds — fine; this is where AI-generated music shines.
  • AAA with bespoke composer themes — no. Hire a composer.
  • Anything vocal-heavy — instrumentals are stronger; vocals are still the weakest link across all current models.
What about copyright on the generated music?

Each generation backend has its own copyright / ownership terms — read them before shipping. The engine's separation, slicing, and runtime code are MIT and make no claim on the audio you put through them.
